# ארכיטקטורת המערכת ומבנה מודולרי 🏗️

שעשועון הטריוויה הקבוצתי **"מי אמר מה?"** בנוי כאפליקציית דף יחיד (SPA) מודרנית המבוססת על **React (TypeScript)** ו-**Vite**. האפליקציה מחולקת לשני מסכים מרכזיים המסונכרנים בזמן אמת: **שלט המנחה (Remote Controller)** ו**מסך ההקרנה (TV Projector View)**.

---

## 📂 עץ קובצי הפרויקט

המערכת מחולקת בצורה מודולרית ונקייה המפרידה בין תשתית הנתונים, הממשק המנהלי (השלט והאשף) וממשק ההקרנה:

```text
/src
├── /components
│   ├── /admin
│   │   ├── AdminContext.tsx       # ניהול מצב גלובלי (React Context) של שלט המנחה
│   │   ├── AdminWizard.tsx        # אשף יצירת/עריכת חדר ב-6 שלבים
│   │   ├── ControlTab.tsx         # לשונית השליטה והפעלת המשחק בשלט
│   │   ├── ImportExportTab.tsx    # ייבוא/ייצוא שאלות ומשתתפים (Excel/JSON)
│   │   ├── MembersTab.tsx         # ניהול והוספת משתתפים
│   │   ├── QuestionsTab.tsx       # ניהול שאלות, ציטוטים ועריכתם בלייב
│   │   ├── SettingsTab.tsx        # הגדרות חדר, טיימרים ושיטות הקראה
│   │   └── StatsTab.tsx           # סטטיסטיקות וניתוח נתוני המשחק
│   ├── AdminView.tsx              # המכולה המנהלית הראשית (בורר בין אשף לשלט)
│   └── GameView.tsx               # מסך ההקרנה הראשי (TV View) התלוי בסנכרון ענן
├── /utils
│   ├── audioHelper.ts             # סינתזה וניהול אפקטים קוליים ומוזיקת מתח (Web Audio API)
│   ├── db.ts                      # שכבת נתונים מקומית (LocalStorage) וטיפוסי נתונים
│   ├── excelHelper.ts             # ניתוח וייבוא קובצי אקסל של שאלות/משתתפים
│   ├── firebase.ts                # אתחול וחיבור ל-Firebase Client SDK
│   ├── imageHelper.ts             # עיבוד וכיווץ תמונות למניעת חריגות נפח
│   └── sync.ts                    # מודול סנכרון והודעות בזמן אמת מול Firebase RTDB
├── App.tsx                        # נקודת כניסה, ניתוב ומסכי בית/בורר מצבים
├── index.css                      # עיצוב גלובלי, משתני CSS ואנימציות
└── main.tsx                       # רינדור האפליקציה ב-DOM
```

## 🔄 ניהול מצב (State Management)

### 1. קונטקסט הניהול (AdminContext)
ממשק הניהול של השלט עושה שימוש ב-React Context (`AdminContext`) על מנת לחלוק פונקציות ניהוליות ומידע בין כל הטאבים של השלט והאשף. הקונטקסט חושף:
- **מצב האשף הלוקאלי**: שם מנחה, מספר מתמודדים, רשימת משתתפים ושאלות שנמצאים בתהליך הגדרה.
- **פונקציות סנכרון**: יצירת חדר בענן, איפוס משחק, הפעלה/השהיה וסנכרון מלא של בסיס הנתונים ל-Firebase.
- **שליטה בשחקנים ושאלות**: פונקציות להוספה, מחיקה ועריכה בזמן אמת של משתתפים וציטוטים.

> [!NOTE]
> **הערת ארכיטקטורה (God Object Pattern)**: 
> רכיב ה-`AdminContext` משמש כיום כמרכז הכובד הראשי של המערכת המנהלית ("God Object"). למרות קיומם של מודולים נפרדים ללשוניות השונות, ניהול המצב ופונקציות העדכון המרכזיות מרוכזות כולן בתוכו כדי להבטיח עקביות ולמנוע מרוצי עדכונים (Race Conditions) מול Firebase RTDB. מומלץ להשאיר מבנה זה כפי שהוא לשם שמירה על יציבות סנכרון הנתונים, אך בעתיד ניתן להעביר לוגיקות משנה (כמו אבחון או סטטיסטיקה) לקבצי עזר חיצוניים.

### 2. שימוש בטיוטות מקומיות (Wizard Drafts)
כדי למנוע אובדן מידע יקר ערך בעת עריכת חדר או מעבר בין מסכים במכשירים ניידים, המערכת שומרת טיוטה לוקאלית תחת המפתח:
`wizard_draft_{ROOM_CODE}` ב-LocalStorage.
- בכל מעבר שלב באשף או שינוי הגדרה, הנתונים נשמרים מקומית.
- בעת טעינת החדר מחדש, הנתונים משוחזרים מהטיוטה המקומית במידה והיא קיימת, או נטענים ישירות מ-Firebase במידה ולא קיימת טיוטה מקומית, תוך הבטחת סנכרון מלא.

### 3. מניעת יציאה בטעות (Back Navigation Guard)
בשלט המנחה מוטמעת האזנה לאירוע `beforeunload` של הדפדפן. במובייל, שימוש במחוות ניווט לאחור (Swipe to go back) או לחיצה על כפתור "חזור" של המכשיר מקפיצה התרעה השואלת את המנחה האם הוא בטוח ברצונו לעזוב את המשחק הפעיל, על מנת למנוע ניתוק של המשחק בעיצומו.
